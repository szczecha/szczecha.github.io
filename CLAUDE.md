## Project Overview

**szczecha.github.io** is a personal portfolio and blog site for QA Engineer / API testing specialist. Built with [Astro](https://astro.build) (v5.8.0), [Tailwind CSS v4](https://tailwindcss.com), [MDX](https://mdxjs.com), and tested with [Playwright](https://playwright.dev).

The site showcases projects, domain knowledge, and technical blog posts focused on API testing, test architecture, observability, and AI.

## Common Development Commands

```bash
npm run dev              # Start dev server at http://localhost:4321
npm run build           # Build for production
npm run preview         # Preview production build locally
npm test                # Run all Playwright tests
npm run test:ui         # Run tests with interactive UI
npm run test:a11y       # Run accessibility tests only
npm run test:report     # View HTML test report
npm run astro           # Run astro CLI commands directly
```

## Project Structure

### Content & Collections
- **`src/content/blog/`** — Blog posts in Markdown/MDX format, loaded via glob pattern
- **`src/content.config.ts`** — Content collection schema; blog posts require: `title`, `description`, `pubDate`, optional `updatedDate`, `heroImage`, `featured`

### Pages & Routes
- **`src/pages/`** — Astro pages; defines routes (index, about, blog listing, 404, newsletter-signed-up)
- **`src/layouts/`** — Reusable layout templates (`BaseLayout.astro`, `ArticleLayout.astro`)
- **`src/components/`** — Astro components
  - `Header.astro`, `Footer.astro`, `LightDarkToggle.astro` — Global UI
  - `articles/ArticleCard.astro`, `FeaturedArticles.astro` — Blog-specific components
  - `Image.astro` — Custom responsive image component
  - `Timeline.astro`, `Projects.astro`, `DomainKnowledge.astro` — Page-specific sections

### Styles & Config
- **`src/styles/`** — Global CSS (used with Tailwind)
- **`astro.config.mjs`** — Astro configuration; enables MDX, sitemap, icon integration; disables image optimization in dev mode
- **`tailwind.config.*`** — Tailwind CSS configuration (Tailwind v4, so config may be in astro.config.mjs)
- **`.prettierrc`** — Uses Prettier with `prettier-plugin-astro` (tabs, width 4)

### Testing
- **`tests/`** — Playwright E2E and accessibility tests
  - `blog.spec.ts` — Blog listing & article page tests
  - `home.spec.ts`, `about.spec.ts` — Page-specific tests
  - `navigation.spec.ts` — Navigation flow tests
  - `accessibility.spec.ts` — a11y compliance (uses @axe-core/playwright)
  - `theme.spec.ts` — Light/dark mode toggle tests
- **`playwright.config.ts`** — Playwright config; baseURL=`http://localhost:4321`, auto-starts dev server, chromium only

### Images & Assets
- **`src/images/`** — Static images (hero images for about page, etc.)
- **`public/`** — Static public assets

## Key Architecture Patterns

### Dark Mode
- Implemented via `LightDarkToggle.astro` component
- Uses CSS classes for theme switching (likely `dark:` Tailwind utilities)
- Tested in `theme.spec.ts`

### Content Metadata
- Blog posts use frontmatter schema validation (Zod)
- `featured` boolean allows flagging posts for display on home page
- Hero images are optional; `OG:image` meta tag uses hero image if present

### TypeScript Configuration
- Extends `astro/tsconfigs/strict` with `strictNullChecks: true`
- `.astro/types.d.ts` auto-generated by Astro

### Image Handling
- Production: Uses Sharp for optimization
- Development: Image optimization disabled (allows StackBlitz/cloud environments to work)
- Use `Image.astro` component for responsive images with proper alt text

## Blog Writing Standards

**Apply these standards when drafting or editing blog posts** (see `/src/content/blog/` examples for reference).

### Structure
Posts follow an arc: problem → origin → growth → impact

1. Meta block (HTML) at top — Role, Company, Timeline, Type
2. Tag pills — monospace tech/domain tags
3. Single CTA link near top (e.g., "View Collection →")
4. **The Problem** — what gap existed
5. **How It Started / Origin** — personal, concrete trigger
6. **Evolution** — subsections showing growth over time
7. **Internal Adoption** — how it spread in the organization first
8. **Going Public / Scaling** — external reach milestone
9. **Demo** — video or screenshots showing, not just telling
10. **Impact** — concrete, terse bullet points of outcomes

Not every post needs all sections, but problem → origin → growth → impact arc is essential.

### Voice & Tone
- First person throughout ("I kept sending...", "I created...")
- Direct and personal — no corporate "we" unless genuinely team effort
- Confident but outcomes-focused, not boastful
- Practical and grounded — assume domain knowledge; no warm-up
- Honest about scrappy origins before becoming official

### Headers
- Short, action-oriented ("How It Started", "Going Public", "Internal Adoption")
- Subheadings show evolution within sections

### Content Discipline
- Tight, no padding — each section earns its place
- Typical length: 400–700 words of prose (excluding code/visuals)
- Visuals essential for tool/product posts — show actual UI
- Lists used sparingly; only when items are parallel
- No lists as narrative substitutes

### Impact Section Format
- Always bullet points
- Concrete and terse — no explanatory prose
- Mix quantitative and qualitative outcomes
- Can include ongoing/living notes (e.g., community maintenance)

## Testing Guidelines

### Running Tests
- Always start with `npm run dev` before running tests; Playwright auto-starts it if not already running
- Use `npm run test:ui` for interactive debugging
- Use `npm run test:a11y` for accessibility audits only
- Check `npm run test:report` after test runs to review failures

### Test Structure Patterns
Tests use Playwright's `test.describe()` for grouping and `test.beforeEach()` for setup. See `blog.spec.ts` for example patterns:
- Validate page headings, titles, and content presence
- Test navigation and interaction (clicking cards, links)
- Verify accessibility attributes (`alt` text, external link `rel` tags)
- Check meta tags (OG:image, etc.)
- Test dark mode appearance if relevant

### Accessibility
- External links must have `rel="noopener noreferrer"`
- Images must have meaningful `alt` text
- Use semantic HTML (h1, h2, article, time, nav)
- Test with `npm run test:a11y`

## Development Notes

### Image Optimization in Dev
The config disables image optimization in development mode (when `NODE_ENV === "development"`). This allows the project to run in constrained environments like StackBlitz. Production builds use Sharp.

### Content Loader
Blog posts are loaded via glob pattern (`**/*.{md,mdx}`), so files are discovered automatically. No manual manifest needed.

### Trailing Slashes
Config sets `trailingSlash: "never"`, so routes are `/blog` not `/blog/`.

### Environment
- Site URL: https://szczecha.github.io (for sitemap, RSS, metadata generation)
- Dev base URL: http://localhost:4321 (Playwright tests use this)

## TypeScript

- All Astro pages are `.astro` files (not `.tsx`); components can be Astro or client-side (React/Preact if added)
- Use TypeScript in `.astro` files within the frontmatter section (top, between `---`)
